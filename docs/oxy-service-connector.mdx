# Oxy service connector

Clarity integrates with the rest of the Oxy ecosystem — and with future third-party services — through a **manifest-driven protocol**. A service registers an `OxyService` document declaring its tools, events, and optional context endpoint. Clarity auto-discovers the manifest at chat time, generates AI-SDK `tool()` wrappers with Zod schemas, and forwards the user's OxyHQ JWT to the service API.

**Onboarding a new service requires zero changes to Clarity's codebase** — only an `OxyService` document insert.

## How it works

1. **Service manifests** live in MongoDB (`OxyService` model). Each manifest defines the service's identity, tools, events, and optional context endpoint.
2. **`buildOxyServiceTools()`** reads manifests at chat time, converts the JSON Schemas to Zod via `jsonSchemaToZod()`, and produces AI-SDK `tool()` wrappers. The wrappers call the service API with the user's JWT.
3. **Events** flow back via `POST /webhooks/oxy/:serviceId` with HMAC signature verification. Events trigger notifications, context-graph updates, or autonomous agent sessions.
4. **Context endpoints** (optional) return a brief, user-scoped summary that Clarity injects into the system prompt at chat start.

## Manifest shape

`OxyService` documents follow the schema in [`packages/backend/src/models/oxy-service.ts`](https://github.com/OxyHQ/Clarity/blob/main/packages/backend/src/models/oxy-service.ts):

```ts
{
  serviceId: 'oxy-notes',          // unique kebab-case id
  displayName: 'Notes',
  baseUrl: 'https://notes.oxy.so/api',
  webhookSecret: '...',            // HMAC for inbound events
  contextEndpoint: '/clarity/context',  // optional, GET
  tools: [
    {
      name: 'searchNotes',
      description: 'Search the user\'s notes by query string.',
      schema: {                    // JSON Schema, converted to Zod at runtime
        type: 'object',
        properties: {
          query: { type: 'string' },
          limit: { type: 'integer', default: 10 },
        },
        required: ['query'],
      },
      endpoint: {
        method: 'GET',
        path: '/notes/search',
      },
    },
  ],
  events: [
    { name: 'note.created', schema: {...} },
    { name: 'note.shared',  schema: {...} },
  ],
}
```

Tool names exposed to the model are namespaced: `oxy_{serviceId}__{toolName}`. For example, the tool above is called `oxy_oxy-notes__searchNotes` from the AI's perspective. The namespace prevents collisions across services and makes the source obvious in `clarity.tool_result` events.

## Chat-time integration

At chat start, [`packages/backend/src/routes/v1/chat-completions.ts`](https://github.com/OxyHQ/Clarity/blob/main/packages/backend/src/routes/v1/chat-completions.ts) calls `buildOxyServiceTools()` to merge service tools into the model's toolset:

```ts
import { buildOxyServiceTools } from '@/lib/tools/oxy-services';

const oxyTools = await buildOxyServiceTools({
  userId,
  accessToken: req.accessToken,
  signal,
});

const tools = {
  ...callerSuppliedTools,
  ...integrationTools,
  ...mcpTools,
  ...oxyTools,
};
```

`buildOxyServiceTools()` uses the same `safeExecute()` + cache pattern as `integrations.ts` and `mcp.ts`. Service tool calls are wrapped in error handling that surfaces sanitised messages to the model and writes raw errors to the routing log.

### Authentication

For first-party Oxy services, the user's OxyHQ JWT is forwarded as `Authorization: Bearer <jwt>`. The receiving service validates the JWT against the shared OxyHQ session table — no OAuth dance, no consent screen.

For third-party services in the future, the manifest can declare an OAuth client and a consent scope set; Clarity will mediate the OAuth flow before enabling the service per-user.

## Context endpoints

Manifests can declare an optional `contextEndpoint` (a `GET` path that takes the user's JWT and returns a short JSON summary). Clarity calls it at chat start and injects the response into the system prompt as a context fragment:

```ts
const fragment = await getOxyServicePromptFragment(service, userId, accessToken);
// → "User has 3 unread emails, 1 calendar event today, …"
```

Use this sparingly — every fragment costs tokens. Services should return only **decision-relevant** state, not raw inventories.

## Event webhooks

Services emit events back to Clarity by posting to:

```
POST /webhooks/oxy/:serviceId
X-Oxy-Signature: t=<timestamp>,v1=<hmac>
Content-Type: application/json

{
  "eventId": "evt_01HX...",      // idempotency key
  "name": "note.shared",
  "userId": "...",
  "payload": { ... },
  "occurredAt": "2026-04-10T12:34:56Z"
}
```

Runtime behavior (`packages/backend/src/routes/oxy-service-events.ts`):

- **Signature verification** — HMAC over `timestamp + body` using `webhookSecret`.
- **Idempotency** — duplicate `eventId` values are dropped silently.
- **Event log** — every accepted event is persisted to `OxyServiceEventLog` for replay and debugging.
- **Routing** — events are dispatched to:
  - Notification rules (push, in-app, email)
  - Context-graph updates (`ContextNode` / `ContextEdge` upserts)
  - Autonomous agent sessions (create `AgentSession` first, then queue for execution)
- **Fallback** — if autonomous execution fails, the runtime falls back to a notification so the user is never silently dropped.

## Adding a new service

```ts
// One-time MongoDB insert (e.g. via seed script)
await OxyService.create({
  serviceId: 'oxy-notes',
  displayName: 'Notes',
  baseUrl: 'https://notes.oxy.so/api',
  webhookSecret: process.env.NOTES_WEBHOOK_SECRET,
  tools: [
    {
      name: 'searchNotes',
      description: 'Search the user\'s notes.',
      schema: { ... },
      endpoint: { method: 'GET', path: '/notes/search' },
    },
  ],
});
```

That's it. The next chat turn includes `oxy_oxy-notes__searchNotes` as a tool the model can call — **no Clarity-side changes required.**

To remove a service, delete (or set `enabled: false` on) the manifest. Running conversations finish with the previous toolset; new conversations see the updated catalogue.

## Patterns to follow

When extending this system:

- Use the same `safeExecute()` + per-user cache pattern as [`integrations.ts`](https://github.com/OxyHQ/Clarity/blob/main/packages/backend/src/lib/tools/integrations.ts) and [`mcp.ts`](https://github.com/OxyHQ/Clarity/blob/main/packages/backend/src/lib/tools/mcp.ts).
- Use `jsonSchemaToZod()` from `mcp-schema.ts` for runtime schema conversion — keep the manifest format JSON Schema for transport.
- Forward `req.accessToken` (the user's OxyHQ JWT) for first-party services. Do **not** invent a new auth mechanism.
- Namespace tool ids as `oxy_{serviceId}__{toolName}` everywhere — it's load-bearing for tool-call attribution.
- Treat tool descriptions as prompts: the model picks tools based on description text. Make them precise.
