# Model abstraction

Clarity is multi-provider on the backend (OpenAI, Anthropic, Google, Groq, DeepSeek, xAI, Mistral, …) but exposes a **single Clarity-branded surface** to users and developers. Provider names, provider model IDs, and routing decisions are strictly internal — they never appear in API responses, UI strings, error messages, SEO metadata, or analytics dashboards.

This page documents the abstraction: the public model tiers, how routing and fallback work, and the invariants every contributor must preserve.

## Public catalogue

Clarity ships a small, named tier set. The complete list (with token limits, credit multipliers, and category flags) lives in [`packages/backend/src/internal/providers/lib/clarity-models.ts`](https://github.com/OxyHQ/Clarity/blob/main/packages/backend/src/internal/providers/lib/clarity-models.ts).

| Model id | Tier | Use case |
|----------|------|----------|
| `clarity-fast` | `fast` | Snappy responses for simple, low-credit tasks |
| `clarity-v1` | `v1` | Default; everyday answers with web search + citations |
| `clarity-pro` | `pro` | Higher-quality reasoning for complex prompts |
| `clarity-thinking` | `thinking` | Long-horizon multi-step reasoning |
| `clarity-pro-max` | `pro-max` | Top-tier multimodal pro workloads |
| `clarity-v1-codea` | `v1-codea` | Codea-surface variant tuned for code |
| `clarity-v1-cowork` | `v1-cowork` | Cowork-surface variant |
| `clarity-v1-browser` | `v1-browser` | Browser-tool workflows |
| `clarity-v1-vision` | `v1-vision` | Image input + reasoning |
| `clarity-v1-audio` | `v1-audio` | Audio understanding |
| `clarity-v1-tts` | `v1-tts` | Text-to-speech |
| `clarity-v1-image` | `v1-image` | Image generation |

Each `ClarityModel` carries a `creditMultiplier`, `maxTokens`, and capability flags (`supportsTools`, `supportsVision`). The `GET /v1/models` endpoint returns these descriptors only — never the underlying mapping.

## Routing and fallback

Each Clarity tier maps to **one or more** provider models, ranked by priority and quality. The routing table lives in `packages/backend/src/internal/providers/lib/generate-model-mappings.ts` and is rebuilt at startup.

Resolution order at chat time:

1. **Tier resolution** — `resolveModel(modelId)` looks up the tier, validates the request's required capabilities (vision, tools, max tokens, etc.).
2. **Mapping selection** — `getModelMappingsForTier(tier)` returns the ranked candidate provider models for that tier, filtered by:
   - Health (`provider-health.ts` — circuit breaker per provider)
   - Cost budget (`pricingTier`: `free` → `freemium` → `paid`)
   - Required capabilities (the request's `tools`, `vision`, etc.)
3. **Failover** — the runtime tries candidates in order. Transient errors (5xx, network, rate limit) advance to the next candidate and emit a `clarity.model_switch` event.
4. **Non-retryable errors** (`format`, `content_filter`) abort immediately — they're tier-level issues, not provider-level, so retrying is pointless.

The user always sees the original `clarity-*` model id in the response — even if four providers were tried.

## Sanitisation invariants

Three layers ensure provider names never leak:

### 1. Error messages

Every user-facing error path calls `sanitizeMessage()` from [`packages/backend/src/lib/errors/sanitize.ts`](https://github.com/OxyHQ/Clarity/blob/main/packages/backend/src/lib/errors/sanitize.ts). It strips known provider names, provider model IDs, and provider-specific error codes, replacing them with neutral language.

```ts
import { sanitizeMessage } from '@/lib/errors/sanitize';

return res.status(429).json({
  error: {
    message: sanitizeMessage(rawErr.message),
    type: 'rate_limit_exceeded',
  },
});
```

### 2. Analytics and usage display

When rendering analytics (top models, per-conversation usage), the API resolves each underlying `provider:modelId` back to a Clarity model via `getClarityModel()`. Rows that **cannot be resolved** to a Clarity tier are skipped — never displayed as provider raw IDs.

### 3. Public model API

`GET /v1/models` and `GET /v1/models/:modelId` return only the `ClarityModel` shape. The `owned_by` field is hard-coded to `"clarity"`. No provider hints, no fallback chains, no priority scores.

## What never to do

These rules apply to **all** contributors:

- Never show provider names (`OpenAI`, `Anthropic`, `Google`, `Groq`, etc.) in UI strings, API responses, error text, SEO metadata, or documentation.
- Never show provider model IDs (`gpt-*`, `claude-*`, `gemini-*`, etc.) to users.
- Never reference specific provider models in feature descriptions or marketing copy.
- Never let an unsanitised provider error string reach the response body.
- Never log provider response bodies at `info` level — restrict to `debug` in development only.

## What to always do

- Use Clarity model ids: `clarity-v1`, `clarity-fast`, `clarity-pro`, `clarity-thinking`.
- Run user-facing error strings through `sanitizeMessage()`.
- Resolve usage events to Clarity tiers via `getClarityModel()`; drop unresolvable entries.
- Restrict all provider routes to the **internal subnet** — `packages/backend/src/internal/` is CORS-locked and not exposed publicly.

## Adding a new Clarity tier

1. Define the entry in `CLARITY_MODELS` (id, tier, capabilities, credit multiplier).
2. Add mappings in `generate-model-mappings.ts` (priority-ordered provider candidates).
3. Ensure `provider-names.ts` covers any new internal provider, with sanitisation entries in `sanitize.ts`.
4. Add tests covering capability fall-through (vision/audio/voice variants) and the sanitisation path.
5. Update analytics resolvers if the new tier has a unique category.

No frontend strings change — the catalogue is fetched live via `GET /v1/models`.
